# AGENTS.md - Agent Coding Guidelines for GNdb

## Project Overview

GNdb is a CLI tool for managing the lifecycle of a PostgreSQL database
for GNverifier (a scientific name verification service). It enables
users to set up and maintain local GNverifier instances with custom
biodiversity data sources.

The tool handles:
- Database schema creation and migration using GORM
- Data population from SFGA (Species File Group Archive) files
- Database optimization for fast name verification and synonym resolution

## Branches

- `main` branch is going through ground-up refactoring.
- `spec-kit` branch contains original code generated by Claude using
  Spec-kit; its code is being transferred to `main` step by step.

## Architecture

### Application Directories

GNdb uses standard XDG-compliant directories:

- **Config** (`~/.config/gndb/`): `config.yaml` and `sources.yaml`.
  Created automatically on first run.
- **Cache** (`~/.cache/gndb/`): Downloaded SFGA files and temporary
  data. Safe to delete; recreated as needed.
- **Logs** (`~/.local/share/gndb/logs/`): `gndb.log` (structured JSON
  or text). Fresh file per run; bootstrap logs preserved during
  logger reconfiguration.

All directories are created during bootstrap (`cmd/root.go`).

### Key Data Flow

1. **Configuration Loading** (`pkg/config`):
   - Loads from `~/.config/gndb/gndb.yaml`
   - Overrides with environment variables (`GNDB_*` prefix)
   - Final overrides from CLI flags
   - Precedence: flags > env vars > config file > defaults

2. **Database Schema** (`gndb.SchemaManager`):
   - GORM AutoMigrate for idempotent schema creation/migration
   - Models in `pkg/schema/models.go`
   - Key tables: DataSource, NameString, Canonical, NameStringIndex,
     VernacularString, Word

3. **Data Population** (`gndb.Populator`):
   - Reads `sources.yaml`; connects to SFGA SQLite files via sflib
   - Transforms data to PostgreSQL schema
   - Bulk inserts using pgx CopyFrom for performance

4. **Optimization** (`gndb.Optimizer`):
   - Creates indexes for fast lookups
   - Builds materialized views and denormalized tables
   - Always rebuilds from scratch to apply algorithm improvements

### Database Schema Highlights

Aligned with gnidump for gnverifier compatibility:

- **DataSource**: Dataset metadata (ID, title, version, URLs, curation)
- **NameString**: Scientific name-strings with parsing metadata
  (UUID-based IDs, canonical forms, cardinality)
- **Canonical/CanonicalFull/CanonicalStem**: Name representations for
  matching
- **NameStringIndex**: Links name-strings to data sources with
  taxonomic metadata (rank, status, classification)
- **Word/WordNameString**: Word-level indexes for fuzzy matching
- **VernacularString/VernacularStringIndex**: Common names in multiple
  languages

### SFGA Data Sources

Configured in `sources.yaml` (`~/.config/gndb/sources.yaml`):
- IDs < 1000 = official, IDs >= 1000 = custom
- `parent` is a directory or HTTP URL containing SFGA SQLite files
- Metadata fields override SFGA `col__*` fields where needed
- `prefer_flat_classification: true` enables flat (non-hierarchical)
  classification import for that source; auto-detected at populate time

### CLI Interface

Cobra and Viper are used as the CLI framework (`cmd/` directory).
`main.go` has only one reference to `cmd`. The `cmd` module is
otherwise self-contained. Subcommands: `create`, `migrate`, `populate`,
`optimize`.

## Code Style Guidelines

### General Principles

- Follow standard Go formatting (enforced by `go fmt`)
- Code should fit within 80 columns unless absolutely necessary
- Use golangci-lint for linting - **all lint issues must be fixed**
- Run `go mod tidy` before finalizing any changes

### Imports

- Use standard Go import ordering with blank lines between groups:
  1. Standard library
  2. External packages (github.com, etc.)
  3. Internal packages (github.com/gnames/gndb/...)
- Always use explicit imports, no unnamed imports
- Example:
```go
import (
    "context"
    "fmt"

    "github.com/gnames/gn"
    "github.com/spf13/cobra"

    "github.com/gnames/gndb/internal/iofs"
    "github.com/gnames/gndb/pkg/config"
)
```

### Naming Conventions

- **Variables**: camelCase (e.g., `homeDir`, `cfg`, `opts`)
- **Constants**: PascalCase for exported, camelCase for unexported
- **Functions**: PascalCase for exported, camelCase for unexported
- **Types/Structs**: PascalCase for exported, camelCase for unexported
- **Interfaces**: PascalCase, typically with "er" suffix (e.g., `SchemaManager`, `Populator`)
- **Packages**: lowercase, short, descriptive (e.g., `config`, `gndb`, `iofs`)
- **Acronyms**: Keep original casing (e.g., `URL`, `HTML`, but `url` variable)
- **Tests**: `Test` prefix for test functions, describe behavior: `TestGetRootCmd_Exists`

### Types and Declarations

- Use meaningful variable names - avoid single letters except in loops
- Group related declarations together
- Use pointers for large structs or when nil is meaningful
- Use value receivers for simple getters, pointer receivers for methods that modify state
- Keep unexported types private (lowercase struct names)

### Error Handling

- Return errors directly rather than wrapping when context is clear
- Use `gn.Error` struct for user-facing errors with dual output:
  - `Msg`: User-friendly message displayed to STDOUT (supports `<em>` tags for emphasis)
  - `Err`: Detailed error with context written to log file
  - `Code`: Error code from `pkg/errcode/` for programmatic handling
- Use `gn.Info()` for user-facing informational messages
- Use `gn.Warn()` for non-fatal warnings
- Detailed logs written to file by default, not displayed to user
- Example pattern:
```go
if err != nil {
    return &gn.Error{
        Code: errcode.SomeError,
        Msg:  "User message",
        Vars: []any{arg1},
        Err:  fmt.Errorf("detailed context: %w", err),
    }
}
```

### Configuration Pattern

**Core principles:**
- Configuration is immutable after creation via `config.New()`
- **All mutations go through Option functions** — the only way to
  modify Config
- Default config is always valid (no validation needed on creation)
- Option functions validate inputs and reject invalid values with
  `gn.Warn()`; invalid options are silently skipped

**Configuration flow:**
1. `cfg := config.New()` — always valid defaults
2. `cfgViper.ToOptions()` → `cfg.Update(opts)` — load from config.yaml
3. Viper unmarshals env vars directly into cfgViper
4. CLI flags call Option functions directly → `cfg.Update(opts)`

**Key design decisions:**
- **CLI flags → Option functions** (not flags → Config → ToOptions):
  check `cmd.Flags().Changed()` to apply only explicitly set flags
- **`ToOptions()` scope**: persistent settings only (Database, Log,
  JobsNumber). Excludes runtime-only fields (HomeDir, SourceIDs,
  ReleaseVersion/Date, WithFlatClassification)
- **Validation**: happens in Option functions via `isValidString()`,
  `isValidInt()`, `isValidEnum()`. Invalid values trigger `gn.Warn()`;
  config keeps the previous valid value

```go
cfg := config.New()
cfg.Update(cfgViper.ToOptions())
cfg.Update([]config.Option{config.OptHomeDir(homeDir)})
```

### Passing Config to Functions

- **Always pass the whole `*config.Config`**, not substructs
- Document what each function actually uses in comments
- Internal implementations can extract substructs locally for
  readability

```go
// Populate imports data from SFGA sources.
// Uses: cfg.Database, cfg.Populate, cfg.JobsNumber, cfg.HomeDir
func (p *populator) Populate(ctx context.Context, cfg *config.Config) error {
    dbCfg := &cfg.Database  // Extract locally if helpful
    // ...
}
```

### Logging

- Use `slog` for structured logging
- Log at appropriate levels: Debug, Info, Warn, Error
- Include relevant context as attributes, not in message
- Early bootstrap uses hardcoded defaults, later reconfigured with user settings

### Database Operations

- Use pgxpool for connection pooling
- Batch operations use configurable batch size (default 50,000)
- Use pgx CopyFrom for bulk inserts (faster than individual inserts)
- Use GORM for schema migrations, pgx for performance-critical operations

### Testing Conventions

- Use `testing.Short()` to skip integration tests in unit test mode
- Integration tests require PostgreSQL database (gndb_test)
- Use testdata/ directory for fixtures
- Table-driven tests with subtests for readability
- Use testify/assert and testify/require
- Test naming: `Test<FunctionName>_<WhatIsTested>`

### Comments

- Use full sentences with periods for exported functions/types
- Document what each function uses in comments for config parameters
- Explain "why", not just "what"

### Interface-Driven Design

- Define functionality through interfaces in pkg/ (public API)
- Concrete implementations should be private (lowercase struct names)
- Only the constructor function should be exported

```go
// pgxOperator implements db.Operator (private struct)
type pgxOperator struct {
    pool *pgxpool.Pool
}

// NewPgxOperator creates a new operator (public constructor)
func NewPgxOperator() db.Operator {
    return &pgxOperator{}
}
```

### Directory Structure

- **cmd/**: CLI command definitions using Cobra
- **pkg/**: Public API packages (importable by external code)
  - `config/`: Configuration management
  - `db/`: Database operator interface
  - `gndb/`: Core interfaces and version info
  - `schema/`: GORM models
  - `sources/`: SFGA data source configuration
- **internal/**: Internal implementation (not importable)
  - `iofs/`: File system operations

## Environment Variables

Only persistent configuration fields (those in `ToOptions()`) are
available as environment variables. Runtime-only fields (HomeDir,
SourceIDs, ReleaseVersion/Date, WithFlatClassification) are CLI-only.

```bash
# Persistent fields — available as env vars
GNDB_DATABASE_HOST=localhost
GNDB_DATABASE_PORT=5432
GNDB_DATABASE_USER=postgres
GNDB_DATABASE_PASSWORD=secret
GNDB_DATABASE_DATABASE=gnames
GNDB_DATABASE_SSL_MODE=disable
GNDB_DATABASE_BATCH_SIZE=50000
GNDB_LOG_LEVEL=info
GNDB_LOG_FORMAT=json
GNDB_LOG_DESTINATION=file
GNDB_JOBS_NUMBER=8
```

**Key alignment:** `ToOptions()` fields = config.yaml fields = env var
fields. This ensures consistency across all configuration sources.

## Human Development Oriented Style

- Implement functions with the human reader in mind.
- Code must be easy to understand; prefer clarity over cleverness.
- Documentation aims to be concise and clear.
- Three similar lines of code is better than a premature abstraction.
